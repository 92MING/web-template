# Mixed-format chunking fixture

The following text intentionally mixes HTML, Markdown tables, JSON, Python code, shell snippets, and bilingual commentary so the splitter can be stress-tested under realistic conditions.

<section class="overview">
  <h2>Overview</h2>
  <p>系统需要把复杂内容切成长度均衡、语义尽量完整的块，而不是把 HTML tag、JSON key、Markdown table row 或代码 token 从中间硬切断。</p>
  <p>When the content contains multiple formats, line-based splitting alone is not enough, because a single logical section may contain prose, examples, table rows, and nested structured data.</p>
</section>

## Markdown table
| component | role | notes |
| --- | --- | --- |
| splitter | recursive segmentation | prefers paragraphs, lines, sentences, phrases |
| cache | sqlite-based store | may skip read/write depending on per-call flags |
| embedding | vector generation | single input should yield one overall vector |

## Python example
```python
async def build_chunks(service, content: str) -> list[dict]:
    chunks = split_text_by_word_count(content, max_word_count=120)
    vectors = await service.embedding([chunk["text"] for chunk in chunks], use_cache=False, save_cache=False)
    return [
        {"offset": chunk["offset"], "word_count": chunk["word_count"], "vector_dims": len(vector)}
        for chunk, vector in zip(chunks, vectors)
    ]
```

## Short FAQ
Q: 如果最后一个 chunk 特别小怎么办？
A: 应该尽量回看前后相邻 chunk，重新平衡边界，而不是简单接受一个只有两三个词的小尾巴。

Q: What if a paragraph is longer than the configured limit?
A: The fallback splitter should still search for word-level boundaries, including Chinese token boundaries, identifiers, numbers, tags, punctuation groups, and code-like tokens.

Final note: This fixture is deliberately noisy. A useful splitter should still produce chunks that stay under the limit, keep offsets correct, and avoid obviously tiny fragments unless the source section itself is tiny.
